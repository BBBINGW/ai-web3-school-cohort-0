## 流程图说明

本图展示了从**用户意图到链上执行**的最小交叉路径，包含三个域：

### 🟦 AI 域（左侧）
| 节点 | 说明 |
|------|------|
| **用户输入** | 用自然语言描述任务（如"发 0.01 ETH 给 Bob"） |
| **AI Agent (LLM)** | 解析用户意图，决定下一步操作 |

### 🟨 交叉区（中间 — AI × Web3 的交汇点）
| 节点 | 说明 |
|------|------|
| **Tool Use** | AI 调用工具查询链上数据、构建交易参数 |
| **人在环确认** | 🔴 用户确认后交易才进入链上执行（HITL 安全机制） |

### 🟩 Web3 域（右侧）
| 节点 | 说明 |
|------|------|
| **链上交易** | 交易广播到网络，矿工/验证者打包确认 |
| **结果返回** | 返回 Tx Hash 和状态变更，AI 解析后反馈给用户 |

### 整体流程

```
用户输入 → AI 解析意图 → Tool Use 构建操作 → 人确认？→ 链上交易 → 结果返回
                                       ↑                    ↑
                                  AI 与 Web3 交叉点      Web3 域执行
```

## 五问说明

### 1. 这个流程解决什么问题？

解决了**"AI 能不能直接操作链上资产"**的安全边界问题。流程明确了从用户自然语言指令到链上交易执行之间的每一层转换：AI 只负责理解和构建操作，**实际执行权始终在用户手上**（通过钱包签名确认）。

### 2. 哪些步骤由 AI / Agent 辅助？

- **AI 解析意图**：将用户的自然语言（如"发 0.01 ETH"）转化为结构化操作
- **Tool Use**：AI 可以查询链上数据（余额、合约状态）、构建交易参数
- **结果解析**：AI 将链上返回的 Tx Hash 和状态变更翻译回用户能理解的语言

### 3. 哪些步骤必须人工确认？

- **人在环（HITL）** 环节：所有涉及链上资产转移或合约调用的操作，**必须经过用户在钱包弹窗中签名确认**
- Agent 不能绕过 MetaMask 自动签名 — 这是架构层面的安全设计，不是技术限制

### 4. 如何验证最终结果？

- **链上验证**：通过区块浏览器（如 Sepolia Etherscan）输入 Tx Hash 查询交易状态、Gas 消耗、区块确认数
- **AI 辅助验证**：AI 可以读取链上数据并解析结果给用户，但最终信任锚点是**区块浏览器的公开数据**，不是 AI 的解读

### 5. 主要风险点是什么？

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **AI 误解意图** | LLM 可能错误解析用户指令（如金额多加一个零） | HITL 确认时用户可以看到完整交易详情 |
| **授权钓鱼** | Agent 可能引导用户对恶意合约签名授权 | 用户需要核实 `to` 地址和 `data` 字段 |
| **盲签** | 用户不看弹窗内容直接点"确认" | 养成每次确认弹窗的习惯 |
| **私钥/助记词泄露** | Agent 或第三方工具可能诱导用户暴露私钥 | 永远不给任何工具私钥或助记词 |
| **AI 幻觉** | LLM 可能编造不存在的合约地址或交易记录 | 以区块浏览器的数据为准，不轻信 AI 的"看起来合理的"结果 |
