## 🛠️ AI Tool Workflow: Notes from Community Experience

Source: Conversation with a fellow AI×Web3 learner / practitioner

### 1. Cursor (CC) + CC Switch + Obsidian Knowledge Base

| Tool | Role |
|------|------|
| **Cursor (CC)** | Best all-around coding agent. Can handle most tasks without manual intervention |
| **CC Switch** | CLI tool to switch API providers/models in Cursor. Connect different backends |
| **Obsidian** | Personal knowledge base vault. Feed everything in, query via Cursor |

Suggested workflow:
1. Use **DeepSeek** to help configure Cursor (fast setup)
2. Install **CC Switch** to connect your own API key
3. Set up **Obsidian vault** as your personal knowledge base
4. Once set up, you never need to manually download anything — Cursor handles it all

### 2. Codex + DeepSeek Hybrid Strategy

| Tool | Strength | Best For |
|------|----------|----------|
| **Codex CLI** | Strategic orchestration | Writing detailed instructions, PDR docs, task breakdowns |
| **DeepSeek** | Code generation | Executing well-specified tasks |

Key insight: **Codex credits are limited** — using it purely for generation burns credits fast and quality drops. Better to use Codex for **instruction orchestration** (writing very detailed specifications) and let DeepSeek execute.

> "如果囫囵吞枣的来一句，它幻觉会超级重" — Chinese models (DeepSeek) need very clear, step-by-step instructions. Vague prompts → heavy hallucination.

Recommended workflow:
- Window 1: **Codex** writes PDR (Product Design / Requirements) docs + detailed task instructions
- Window 2: **Claude Code** switched to DeepSeek backend, executing those instructions

### 3. Lessons Learned

| Lesson | Detail |
|--------|--------|
| 🎯 **Divide the labor** | Orchestration (Codex) ≠ Generation (DeepSeek) — use each for its strength |
| 📝 **Write tight specs** | The more detailed the instruction, the better the Chinese model output |
| 🚫 **Avoid new tech** | Don't use bleeding-edge libraries/APIs with DeepSeek — outdated training data leads to hallucinations |
| 💡 **Two windows** | Have the orchestrator and executor side by side for faster iteration |

### 4. Why This Matters for AI×Web3 School

This workflow insight directly applies to building in the AI×Web3 space:

```text
Good instruction → Good code → Good tools → Good product
                                     ↓
                            Web3 wallet, contracts, Agents
```

The same principle applies to AI Agents in Web3: **the orchestrator (Agent) writes tight specs, the executor (model/contract) carries them out.**
