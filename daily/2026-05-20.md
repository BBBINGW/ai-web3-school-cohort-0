## 📖 Prompt 章节 — 知识点总结

### 核心认知

> **Prompt 是软约束，不是安全边界。** 真正的安全必须由代码、权限、校验和审计来承担。

### Prompt 的组成部分

| 组件 | 说明 |
|------|------|
| **Instruction** | 任务规则 — 角色、目标、禁止行为、输出格式 |
| **Few-shot** | 示例 — 风格/边界很难说清时用，但需跟 eval 一起维护 |
| **Structured Output** | 固定格式输出（JSON/函数参数）— 让后续代码可校验 |
| **Context** | 可信数据和来源边界 |

### Instruction 的四段式写法

1. **任务目标** — 你要做什么
2. **可用输入** — 你能拿到什么信息
3. **禁止行为** — 不能做什么（区分"解释"vs"执行"）
4. **输出格式 + 失败格式** — 正常/不确定时分别输出什么

### 最小实践：交易风险摘要 Prompt

**风险等级定义：**

| 等级 | 条件 | human approval |
|------|------|:---:|
| 🚨 urgent | 直接威胁资金安全（无限授权到不明合约、循环掏空） | ✅ |
| 🔴 high | 显著风险（大额 approve、simulation 部分失败） | ✅ |
| 🟡 medium | 有不确定性需注意 | ✅ |
| 🟢 low | 看起来安全（标准 transfer） | ❌ |

**输出 JSON 字段：** `summary`, `asset_changes`, `permissions_changed`, `risk_level`, `requires_human_approval`, `uncertainties`, `recommended_user_checks`

**三个测试用例：**
1. 普通 transfer → 预期 low
2. 无限授权（已知合约）→ 预期 medium/high
3. 用户意图与交易不匹配 → 预期 urgent

### 安全链路（Prompt 不独自承担）

```
Prompt → Context → Model → Code → Guard/Simulation → Human check
```

每一层分担一部分安全责任，不是"出事了人兜底"。
